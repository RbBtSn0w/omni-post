# 后端重构完善需求质量自检清单 (PR 评审标准)

**目的**: 本清单作为 PR 评审的“前置单元测试”，用于验证 Node 后端重构相关需求的质量、完整性和安全性，确保在进入代码实现前所有业务逻辑无歧义且已闭环。
**创建日期**: 2026-03-10
**功能**: [specs/025-node-backend-refine/spec.md](../spec.md)

## 1. 核心安全性要求 (Security Quality)

- [x] CHK001 - 是否为所有涉及文件读取和 Cookie 下载的接口明确定义了允许访问的“根目录”路径？ [清晰度, 补全, 针对 Spec §FR-001]
- [x] CHK002 - 对于非法路径（含穿越序列 `..`）的拒绝行为，是否定义了统一的 400 错误响应格式？ [一致性, 针对 Spec §FR-001]
- [x] CHK003 - 是否已明确排除对绝对路径（Absolute Path）的支持，以防止逃逸根目录？ [覆盖度, 缺口]

## 2. 架构稳定性与 ESM 兼容 (Architecture Stability)

- [x] CHK004 - 是否有明确规则强制要求所有 TypeScript/JavaScript 内部模块导入必须包含 `.js` 后缀？ [一致性, 针对 Spec §FR-002]
- [x] CHK005 - 需求是否定义了在构建或 Lint 阶段如何验证 ESM 后缀的准确性？ [可衡量性, 针对 Spec §SC-002]
- [x] CHK006 - 是否明确定义了在 ESM 模式下无法直接使用 `__dirname` 的替代方案（如 `import.meta.url`）的需求？ [架构风险, 针对 Spec §SC-002]

## 3. 并发锁与资源调度 (Concurrency & Lifecycle)

- [x] CHK007 - 当 `LockManager` 获取账号锁失败时，需求是否定义了系统的表现行为（例如：直接报错 423、自动重试、还是入队等待）？ [完整性, 缺口, 针对 Spec §FR-003]
- [x] CHK008 - 是否为所有发布任务（包括各种异常分支）定义了“必须释放锁”的生命周期要求？ [健壮性, 针对 Spec §FR-003]
- [x] CHK009 - 对于 SSE (Server-Sent Events) 登录流，是否完整描述了在客户端意外断开、任务超时和任务正常结束等所有场景下的资源回收逻辑？ [覆盖度, 针对 Spec §FR-005]

## 4. 物理上限与业务限制 (Physical Limits & Business Logic)

- [x] CHK010 - **[关键缺口]**: 当达到全局 5 个活跃浏览器的限制时，需求是否定义了新任务的处理逻辑（是立即返回 429 忙碌，还是在后台无限期排队）？ [完整性, 用户问题反馈]
- [x] CHK011 - **[关键缺口]**: 当账号被锁定或并发超限时，返回给用户的错误消息（msg 字段内容）是否已在需求中标准化？ [清晰度, 用户场景]
- [x] CHK012 - 需求是否定义了单个浏览器上下文的超时时间限制，以防止资源无限期占用？ [覆盖度, 针对 Spec §FR-006]

## 5. API 对等与响应规范 (API Parity & Envelope)

- [x] CHK013 - 是否明确定义了在 3 小时验证冷却期内，如果缓存状态失效，`/getValidAccounts` 接口更新数据库的具体触发条件？ [清晰度, 针对 Spec §FR-004]
- [x] CHK014 - 对于 `/postVideoBatch` 等复杂接口在校验失败（非 JSON 数组）时的 `code/msg/data` 结构是否已明确规定？ [一致性, 针对 Spec §FR-007]
- [x] CHK015 - 是否在需求中明确了哪些 Python 原有的“已知缺失字段”在本次重构中必须找回并返回给前端？ [对等性, 用户反馈]

## 6. 环境与可测试性 (Testability)

- [x] CHK016 - 需求是否定义了测试清理时如何识别并强制删除 SQLite 产生的 WAL 模式辅助文件（-wal, -shm）？ [可衡量性, 针对 Spec §FR-008]
- [x] CHK017 - 成功标准中的“100% 账号状态匹配”是否定义了客观的核对方法（例如：通过第三方工具或人工抽检比例）？ [可衡量性, 针对 Spec §SC-004]

## 备注

- 本清单侧重于 **评审门禁 (PR Review Standard)**，旨在发现 `spec.md` 中可能遗漏的边界逻辑和模糊定义。
- 重点关注 Q2 (并发上限业务逻辑) 和 Q3 (已知缺失点)。
