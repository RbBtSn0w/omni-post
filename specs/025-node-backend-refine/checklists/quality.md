# Node 后端重构与 CI 完善需求质量核对清单

**目的**: 本清单作为 PR 评审的“前置单元测试”，旨在验证 Node 后端重构相关需求的编写质量、逻辑完整性及与 Python 原有版本的对等性，确保开发规划具备清晰的业务基准。
**创建日期**: 2026-03-10
**功能**: [specs/025-node-backend-refine/spec.md](../spec.md)

## 1. 响应结构与 API 对等性质量 (Parity Quality)

- [ ] CHK001 - 是否为所有 API 接口（含错误分支）明确规定了统一的 `{ code, msg, data }` 响应信封结构？ [一致性, Spec §FR-007]
- [ ] CHK002 - 当请求发生错误（如账号锁定或路径非法）时，返回的 `code` 数值和 `msg` 文本是否已定义为与 Python 版完全一致？ [清晰度, Spec §FR-001, §FR-003]
- [ ] CHK003 - 针对 `/getValidAccounts` 接口，需求是否量化了“3 小时”冷却期的具体起始计时节点（如 last_validated_at）？ [清晰度, Spec §FR-004]
- [ ] CHK004 - 是否明确规定了在冷却期内，若用户提供 `force=true` 参数时，接口必须执行的操作逻辑？ [完整性, Spec §FR-004]

## 2. 安全性与目录访问需求质量 (Security Quality)

- [ ] CHK005 - 路径穿越防护要求是否明确涵盖了对“绝对路径”及跨盘符尝试的拒绝准则？ [覆盖度, Spec §FR-001]
- [ ] CHK006 - 需求是否显式规定了文件服务所允许访问的“白名单根目录”的具体路径解析规则？ [清晰度, Gap]
- [ ] CHK007 - 是否定义了当安全校验失败（400 错误）时，系统应记录的审计日志内容？ [完整性, Gap]

## 3. 并发锁与资源生命周期需求质量 (Concurrency & Resource Quality)

- [ ] CHK008 - 账号锁冲突时的“立即报错”逻辑，是否定义了对应的业务提示文本？ [清晰度, 澄清决策]
- [ ] CHK009 - 需求是否规定了账号锁必须在所有异常路径（如执行器内部崩溃、超时）下均能安全释放？ [健壮性, Spec §FR-003]
- [ ] CHK010 - 对于 SSE 登录流，需求是否完整描述了在客户端主动断开、网络波动、任务超时等所有场景下的资源清理行为？ [场景覆盖, Spec §FR-005]
- [ ] CHK011 - [关键缺口]: 由于“不设硬性并发上限”，需求是否定义了当系统物理资源（内存/CPU）耗尽导致任务崩溃时的自动恢复或用户告知逻辑？ [物理限制, 澄清决策]

## 4. 自动化 CI 门禁需求质量 (CI/CD Quality)

- [ ] CHK012 - 需求是否明确了 CI 工作流在 ESLint、TSC 或测试失败时的“强制拦截”合并行为？ [清晰度, Spec §FR-009]
- [ ] CHK013 - 是否量化了 CI 构建的效率目标（如必须包含 npm cache 优化）？ [非功能性, 澄清决策]
- [ ] CHK014 - SC-005 中的 CodeQL 扫描结果验证，是否规定了在 CI 失败时的阻塞级别（拦截合并 vs 仅通知）？ [可衡量性, Spec §SC-005]

## 5. 环境稳定性与可测试性质量 (Stability & Testability)

- [ ] CHK015 - 强制补全 `.js` 后缀的要求，是否在需求中规定了其必须覆盖所有类型的内部引用（含相对路径和别名路径）？ [一致性, Spec §FR-002]
- [ ] CHK016 - 测试清理环节中，“递归删除所有 SQLite 辅助文件”的验收标准是否定义了具体的验证方法（如目录状态核对）？ [可衡量性, Spec §FR-008]

## 备注

- 本清单侧重于 **综合评审 (General)**，旨在发现 `spec.md` 中可能遗漏的边界逻辑和模糊定义。
- 特别关注 Q3 (资源耗尽表现) 的需求补全情况。
