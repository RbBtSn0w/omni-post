# API 对等性与重构质量核对清单: Node 后端重构完善

**目的**: 本清单作为 PR 评审的标准，旨在通过验证需求规范（spec.md）的编写质量，确保 Node 后端在功能、响应结构及行为上与 Python 版本完全对等，并消除技术迁移中的歧义。
**创建日期**: 2026-03-10
**功能**: [specs/025-node-backend-refine/spec.md](../spec.md)

## 1. 响应结构与信封对等性 (Response Structure & Parity)

- [ ] CHK001 - 是否为所有 API 接口明确规定了统一的 `{ code, msg, data }` 响应结构？ [一致性, Spec §FR-007]
- [ ] CHK002 - 在错误场景（如路径穿越或账号锁定）下，需求是否定义了与 Python 版本一致的具体 `code` 数值？ [清晰度, Spec §FR-001, §FR-003]
- [ ] CHK003 - 针对 `/getValidAccounts` 和 `/getAccountStatus` 接口，是否显式定义了返回数据中必须包含的 Python 历史缺失字段（如 `last_validated_at`）？ [完整性, Spec §FR-004]
- [ ] CHK004 - 是否明确定义了在 3 小时验证冷却期内，若数据库状态为“失效”，接口是否应返回缓存数据还是触发强制刷新？ [清晰度, Spec §FR-004]

## 2. 行为逻辑与副作用一致性 (Logic & Side-effect Parity)

- [ ] CHK005 - 账号锁冲突时的“立即报错”逻辑，是否定义了对应的业务提示文本（msg 字段内容）？ [清晰度, 澄清决策]
- [ ] CHK006 - 需求是否规定了账号锁必须在所有异常分支（如上传器内部崩溃）下均能安全释放？ [覆盖度, Spec §FR-003]
- [ ] CHK007 - 对于 SSE 登录流，是否定义了服务器在客户端主动断开连接时的资源回收规范？ [场景覆盖, Spec §FR-005]
- [ ] CHK008 - 上传器使用 `BrowserContext` 的要求，是否在需求层面明确禁止了产生任何非共享的全局浏览器实例？ [一致性, Spec §FR-006]

## 3. 安全性与目录访问规范 (Security & Access Quality)

- [ ] CHK009 - 路径穿越防护要求是否涵盖了对“绝对路径”尝试的拒绝策略？ [覆盖度, Spec §FR-001]
- [ ] CHK010 - 是否明确定义了文件服务根目录（`VIDEOS_DIR`, `COOKIES_DIR`）在不同平台下的基准解析规则？ [清晰度, Gap]

## 4. 环境稳定性与可测试性 (Stability & Testability Quality)

- [ ] CHK011 - 强制补全 `.js` 后缀的要求，是否在需求中规定了其必须覆盖所有类型的内部依赖引用？ [一致性, Spec §FR-002]
- [ ] CHK012 - 测试清理环节中，“递归删除所有 SQLite 辅助文件”的要求是否具备可衡量的验收指标（如目录状态检查）？ [可衡量性, Spec §FR-008]
- [ ] CHK013 - 成功标准 SC-004（100% 账号状态匹配）是否定义了客观的核对方法或允许的同步时间差？ [可衡量性, Spec §SC-004]

## 备注

- 本清单主要用于 PR 评审人确认重构需求已充分定义，以避免在开发阶段产生不必要的反复。
- 重点关注与原有 Python 版本行为的“对等性 (Parity)”。
