# 需求质量自检清单: Node 后端重构与 CI 自动化

**目的**: 本清单作为“需求质量的单元测试”，旨在验证 Node 后端重构及 CI 自动化相关需求的质量、完整性和可衡量性，确保开发规划具备清晰的业务准则。
**创建日期**: 2026-03-10
**功能**: [specs/025-node-backend-refine/spec.md](../spec.md)

## 1. 自动化质量保障需求质量 (CI/CD Quality)

- [ ] CHK001 - 是否明确规定了 CI 工作流在 ESLint、TSC 或测试失败时的“强制拦截”行为（PR 门禁）？ [清晰度, 澄清决策, 针对 Spec §FR-009]
- [ ] CHK002 - 需求是否量化了 CI 的反馈速度要求，并明确提及了缓存优化（如 npm cache）？ [非功能性, 澄清决策]
- [ ] CHK003 - 是否明确定义了 CI 的运行环境（如仅限 Ubuntu-latest）以避免环境范围蔓延？ [范围界限, 澄清决策]
- [ ] CHK004 - 针对 SC-005 中的覆盖率报告，是否定义了报告的可见性（如必须评论在 PR 或上传至仪表盘）？ [完整性, 针对 Spec §SC-005]
- [ ] CHK005 - 需求是否规定了 CI 工作流对 ESM 后缀导入的特定检查规则？ [一致性, 针对 Spec §FR-002]

## 2. API 对等性与响应规范 (API Parity Quality)

- [ ] CHK006 - 是否为所有 API 接口（包括错误分支）明确规定了统一的 `{ code, msg, data }` 结构？ [一致性, Spec §FR-007]
- [ ] CHK007 - 在账号锁冲突或路径非法时，返回的业务错误码（msg）是否已在需求中标准化？ [清晰度, 针对 Spec §FR-003, §FR-001]
- [ ] CHK008 - 针对 `/getAccountStatus` 的 3 小时冷却期逻辑，是否明确定义了强制刷新的参数（force=true）行为？ [完整性, 针对 Spec §FR-004]

## 3. 安全性与资源生命周期 (Security & Resource Quality)

- [ ] CHK009 - 路径穿越防护要求是否明确涵盖了对绝对路径和 `..` 序列的拦截准则？ [覆盖度, Spec §FR-001]
- [ ] CHK010 - 对于 SSE 登录流，需求是否完整定义了在客户端主动断开连接时的服务器侧资源回收行为？ [场景覆盖, Spec §FR-005]
- [ ] CHK011 - 账号锁的生命周期是否明确要求必须在 `finally` 块中释放（含执行器内部崩溃场景）？ [健壮性, Spec §FR-003]

## 4. 成功标准的可衡量性 (Measurability)

- [ ] CHK012 - SC-006（CodeQL 0 漏洞）是否定义了具体的验证节点（如合并前检查）？ [可衡量性, Spec §SC-006]
- [ ] CHK013 - 成功标准中的“100% 账号状态匹配”是否定义了客观的核对方法或允许的时间差？ [可衡量性, Spec §SC-004]

## 备注

- 本清单重点关注 **API 对等性** 与 **CI 自动化** 的需求描述完整性。
- 确保所有 [澄清决策] 中确定的门禁策略和性能优化已在 `spec.md` 中有对应体现。
