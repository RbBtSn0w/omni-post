# 功能规范：优化与完善工作空间及 Package 管理

**特性分支**: `028-refine-package-management`
**创建日期**: 2024-05-22
**当前状态**: completed
**输入**: 用户描述: "现在项目空间有些混乱，特别是package的管理多种场景， 更新和完善当前工作空间。"

## 用户场景与测试 *(必填)*

### 用户场景 1 - 统一的开发入口 (优先级: P1)

作为一名新加入项目的开发者，我希望通过一个简单的根目录命令就能完成所有子项目的环境初始化和依赖安装，而不需要分别进入不同的目录去执行命令。

**优先级说明**: 降低入门门槛，确保所有开发者的环境基准一致。

**独立测试**: 验证在一个干净的代码库中，通过根目录的一个命令即可完成所有子应用的 `npm install` 且无错误。

**验收场景**:

1. **假设** 开发者位于项目根目录，**当** 运行 `npm run setup` 时，**那么** 所有 `apps/*` 和 `packages/*` 下的依赖应被正确安装。
2. **假设** 根目录有新的公共包，**当** 执行全局安装时，**那么** 各子应用应能自动链接或识别该公共包。

---

### 用户场景 2 - 多场景下的脚本一致性 (优先级: P1)

作为一名维护者，我希望所有子应用的脚本（如 `dev`、`build`、`test`、`lint`）遵循相同的命名规范，这样我可以在根目录通过 workspace 标志轻松并行执行或批量执行。

**优先级说明**: 提高 CI/CD 效率和本地运维效率。

**独立测试**: 在根目录使用 `--workspaces` 标志运行 `lint` 或 `test` 命令，所有子应用均应按预期执行并返回结果。

**验收场景**:

1. **假设** 存在多个应用，**当** 运行 `npm run lint --workspaces` 时，**那么** 所有的项目都应执行其对应的代码检查。
2. **假设** 某个应用缺少标准脚本，**当** 进行工作空间检查时，**那么** 系统应给出警告或提供默认行为。
3. **假设** 所有子项目定义了 `test`，**当** 运行 `npm run test --workspaces` 时，**那么** 所有子项目测试应被执行并汇总结果。

---

### 用户场景 3 - 依赖项冲突与冗余清理 (优先级: P2)

作为一名架构师，我希望能够清晰地识别并清理子应用中重复安装的公共依赖，将其提升到根目录管理，以减小安装体积并加快构建速度。

**优先级说明**: 优化 Monorepo 的存储和构建性能。

**独立测试**: 检查 `node_modules` 结构，确保公共依赖（如 `typescript`、`eslint`）仅在根目录存在一份副本（Hoisting）。

**验收场景**:

1. **假设** 多个应用使用相同版本的同一依赖，**当** 执行依赖优化后，**那么** 子应用的 `package.json` 应保持简洁，且依赖已提升至根目录。

---

### 边界情况

- **版本不一致**: 当两个子应用由于特殊原因必须使用不同版本的同一个依赖时，系统必须允许该依赖在子项目本地保留，不强制 Hoisting；冲突依赖需在文档中明确列出并说明原因。
- **Python 遗留项**: Python 后端仅作为可选安装项存在，`npm run setup` 中的 Python 安装必须失败不影响 Node.js 工作空间安装，并输出清晰提示。
- **本地包链接**: 当 `packages/cli` 正在开发中时，所有引用必须使用 `workspace:*`，禁止依赖发布版本；若检测到非 workspace 协议，应提示并阻断发布脚本。
- **部分失败**: 任一子项目安装/脚本失败时，根命令必须返回非零退出码并输出失败项目清单。

## 功能需求 *(必填)*

### 功能需求点

- **FR-001**: 系统必须建立统一的根目录 `package.json` 脚本规范，**至少**涵盖 setup, install, dev, build, lint, test, clean。
- **FR-002**: 系统必须利用 npm workspaces 特性，确保本地包（如 `packages/*`）能被各应用自动发现和链接；所有本地依赖必须使用 `workspace:*`。
- **FR-003**: 根目录必须配置全局的 lint 和 format 规则；子项目必须通过共享配置继承（允许差异化扩展但不允许重复安装核心工具链）。
- **FR-004**: 系统必须提供清晰的文档，说明在不同场景（开发、生产、CI）下应执行哪些 Package 管理命令。
- **FR-005**: 系统必须支持一键清理所有工作空间下的临时构建产物和 `node_modules`，并明确保留项（如 `.env*` 和 `data/`）。
- **FR-006**: 所有 `apps/*` 子项目必须为 `private: true`，并遵循命名规范 `@omni-post/[name]`；`packages/*` 必须命名为 `@omni-post/shared-[name]`。

### 规范补充

#### 1. 根脚本清单 (Root Scripts)
根目录必须提供以下脚本别名：
- `setup`: 初始化所有工作空间依赖，并可选初始化 Python 依赖
- `install`: 仅安装 Node.js 工作空间依赖
- `dev`, `build`, `test`, `lint`, `clean`: 对所有 workspaces 执行对应脚本

#### 2. 全局 lint/format 责任划分
- 根目录负责统一的 ESLint/Prettier/TS 基础配置
- 子项目必须通过共享配置继承，不允许在子项目重复定义核心规则集

#### 3. 文档覆盖场景
文档必须覆盖以下场景的命令清单：
- 开发：本地开发、调试与热更新
- 生产：构建与部署前步骤
- CI：批量 lint/test/build 的工作空间命令

#### 4. 清理范围
`clean` 必须删除：`node_modules/`, `dist/`, `build/`, `coverage/`, `.cache/`, `.vite/`。
`clean` 必须保留：`.env*`, `data/`, `logs/`。

### 核心实体

- **Workspace (工作空间)**: 整个 Monorepo 的顶层容器，定义管理规则。
- **Package (包)**: 位于 `apps/` 或 `packages/` 下的独立单元，具有自己的生命周期。
- **Dependency Map (依赖图谱)**: 描述各包之间的引用关系及对外部库的依赖情况。

## 成功指标 *(必填)*

### 可衡量的结果

- **SC-001**: 执行 `npm install` 的总耗时在依赖不变的情况下降低 20%（由于提升了公共依赖）。
  - **测量方法**: 使用同一台机器、同一 Node/npm 版本、清空 `node_modules` 后执行 `time npm install`，取三次平均值作为基线和优化后对比。
- **SC-002**: 在根目录执行一次测试命令即可覆盖 100% 的**Node.js** 子项目测试（`apps/frontend`, `apps/backend-node`, `packages/cli`）；若某子项目缺少 `test` 脚本则视为失败并输出清单。
- **SC-003**: 新开发者从 clone 到启动第一个 dev 模式的时间缩短至 5 分钟以内。
  - **测量方法**: 从干净 clone 到执行 `npm run setup` 与 `npm run dev -w @omni-post/frontend`，记录总耗时。
- **SC-004**: 所有子应用的 `package.json` 脚本命名重合度达到 100%（即所有应用都必须支持 `build` 等标准命令）。

## 依赖与假设

- **工具链边界**: 以 npm workspaces 为唯一支持方式（pnpm 不在本次范围内）。
- **Hoisting 假设**: 默认依赖 hoisting 由 npm 自动处理；如遇冲突依赖，则允许在子项目保留本地版本并在文档中记录。
