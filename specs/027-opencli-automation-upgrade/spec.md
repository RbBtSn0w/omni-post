## 核心业务规则 (补充)

### 1. CLI 路径探测优先级
系统在启动自动化浏览器时，必须遵循以下路径探测优先级：
1. **显式参数**: 用户通过 CLI 命令直接指定的 `--path`。
2. **数据库配置**: 关联账号时在 Web UI 中绑定的 `BrowserProfile` 路径。
3. **系统默认**: 根据操作系统自动推断的 Chrome 默认用户数据目录。

### 2. 文章发布限制
- **单篇字数**: 建议 100,000 字符以内。
- **传输限制**: 后端 API 单次请求最大支持 50MB 数据。
- **媒体资源**: 文章中引用的本地图片必须在发布前通过 `ExplorerService` 提供的钩子自动上传至目标平台的 CDN。

## 用户场景与测试 *(必填)*

### 用户场景 1 - 本地 Chrome 会话复用 (优先级: P1)

作为一名内容创作者，我希望自动化工具能直接使用我本地 Chrome 浏览器中已经登录的会话，这样我就不必每次发布时都扫描二维码或处理复杂的验证码。

**优先级说明**: 这对提升系统稳定性和用户体验至关重要。复用活跃会话能显著降低账号被风控的风险，并消除频繁重新验证的麻烦。

**独立测试**: 验证用户是否可以关联本地浏览器配置文件，并在没有任何交互式登录步骤的情况下成功访问平台的创作者中心。

**验收场景**:

1.  **假设** 用户在本地 Chrome 中已登录抖音，**当** 他们将此配置文件关联到 OmniPost 时，**那么** 系统应能直接跳转到抖音上传页面而无需提示登录。
2.  **假设** 已关联本地配置文件，**当** 本地浏览器中的会话过期时，**那么** OmniPost 应提供清晰的通知并引导用户在本地浏览器中刷新会话。

---

### 用户场景 2 - 多平台文章发布 (优先级: P1)

作为一名多平台作者，我希望将我的 Markdown 文章同时发布到多个博客平台（知乎、掘金等），以便以最少的精力覆盖更广泛的受众。

**优先级说明**: 将产品范围从视频扩展到文章，使其成为真正的“全能”内容发布工具。

**独立测试**: 验证单个 Markdown 文件是否能以正确的格式和元数据（标题、标签）发布到至少两个不同的文章平台。

**验收场景**:

1.  **假设** 有一个 Markdown 文件和标题，**当** 用户选择知乎和掘金作为目标时，**那么** 两个平台都应接收到内容，并保留格式（如加粗、链接、代码块）。
2.  **假设** 有一个定时发布文章的任务，**当** 到达预定时间时，**那么** 系统应自动执行所有选中平台的发布流程。

---

### 用户场景 3 - CLI 优先工作流 (优先级: P2)

作为一名开发者或高级用户，我希望通过命令行界面管理我的发布任务和账号，以便将 OmniPost 集成到我自己的自动化脚本和工作流中。

**优先级说明**: 提高技术用户的效率，支持更好的工具集成，向 OpenCLI 的高效模式看齐。

**独立测试**: 所有核心操作（列出任务、添加账号、触发发布）均可通过终端命令执行，并支持结构化输出。

**验收场景**:

1.  **假设** 系统正在运行，**当** 用户运行 `omni-cli task list --format json` 时，**那么** 他们应收到当前发布任务的有效 JSON 数组。
2.  **假设** 有一个视频文件，**当** 用户通过 CLI 运行发布命令时，**那么** 系统应在后台创建一个新任务并开始执行。

---

### 用户场景 4 - AI 辅助平台探索 (优先级: P3)

作为一名为 OmniPost 添加新平台的开发者，我需要一个工具来分析平台网站，识别关键交互点并建议自动化策略。

**优先级说明**: 显著降低添加新平台的技术门槛和时间成本，提升项目的可扩展性。

**独立测试**: 对新 URL 运行探索工具可生成包含潜在 API 端点和 UI 选择器的结构化报告。

**验收场景**:

1.  **假设** 提供了一个新平台的 URL，**当** 运行探索工具时，**那么** 它应输出已识别的“发布”按钮、“标题”输入框以及潜在的 XHR/API 接口列表。

---

### 边界情况

*   **并发会话访问**: 如果用户和 OmniPost 同时尝试使用同一个本地 Chrome 配置文件会发生什么？系统会捕获文件锁异常并引导用户关闭浏览器。
*   **动态 UI 变化**: 采用声明式适配器，当 UI 变更时，仅需更新 `Adapter` 的选择器配置即可。
*   **混合内容任务**: 系统已扩展 `Task` 模型，支持同一任务内包含不同类型的内容分发。

## 功能需求 *(必填)*

### 功能需求点

*   **FR-001**: 系统必须支持关联和复用本地 Chrome/Edge/Brave 浏览器的配置文件进行身份验证。
*   **FR-002**: 系统必须实现“文章”内容类型及相应的服务，用于管理文本发布。
*   **FR-003**: 系统必须提供专门的 CLI 工具，以便与所有后端服务进行交互。
*   **FR-004**: 系统必须在现有视频平台基础上，增加对知乎、掘金（文章平台）的支持。
*   **FR-005**: 系统必须采用声明式的“适配器”模式来定义平台特定的交互，以便快速更新。
*   **FR-006**: 系统必须具备“平台探索器”能力，用于发现新网站上的 UI 元素和 API 模式。
*   **FR-007**: 系统必须为需要 HTML 的文章平台提供 Markdown 到 HTML 的转换支持。

### 核心实体

*   **账号 (Account)**: 表示社交平台账号。扩展为包含 `session_source` (托管/本地) 和 `browser_profile_id`。
*   **内容 (Content)**: 所有发布的基类。子类型：`Video` (文件、封面) 和 `Article` (Markdown 内容、封面)。
*   **任务 (Task)**: 发布作业。扩展为支持多内容与多平台的映射。
*   **适配器 (Adapter)**: 定义如何针对特定内容类型自动化特定平台的配置或脚本。

## 成功指标 *(必填)*

### 可衡量的结果

*   **SC-001**: 相比当前的扫码方式，使用本地会话复用可将登录相关的任务失败率降低 70%。
*   **SC-002**: 用户能在 3 分钟内的总交互时间内，成功将一篇文章发布到 3 个平台。
*   **SC-003**: 100% 的现有视频发布功能均可通过新 CLI 访问。
*   **SC-004**: 开发者使用探索工具，从零开始完成一个新平台发布脚本的基础开发时间少于 2 小时。
*   **SC-005**: 95% 的 Markdown 格式在所有支持的文章平台上都能准确保留。
